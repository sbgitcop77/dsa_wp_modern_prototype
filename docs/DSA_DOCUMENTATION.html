DSA Modern Prototype

A Next.js 14 (App Router) prototype redesign of thediamondsportsacademy.com — an elite youth baseball training facility in Odenton, MD. This is a UI/UX prototype: all data is mock-generated, there is no real database, payment processing, or email delivery. This document covers every aspect of the project — architecture, design, data, admin panel, and deployment.

🏟 Overview

The Diamond Sports Academy (DSA) is a real business. This repository is a prototype replacement for their existing WordPress site, built to demonstrate a modern UX and admin panel before production development begins.

🔄 Data Flow

There is no API layer between components and data. Mock data files export typed arrays that are imported directly — identical to how a real app would import from a data-fetching hook, but without the async boundary.

ℹ️
All mock data is generated deterministically at module load time using a seeded LCG (Linear Congruential Generator) RNG. The same seed always produces the same data. Date-relative data (bookings, availability, notifications) is generated relative to new Date() so the prototype never goes stale.
// Typical data import pattern
import { MOCK_BOOKINGS } from "@/data/mock/bookings";
import { MOCK_CUSTOMERS } from "@/data/mock/customers";

// Server components: direct import (no useEffect)
// Client components: useState initialized with the import
const [bookings, setBookings] = useState(MOCK_BOOKINGS);

State Management Pattern

  • Pages are server components by default; only components requiring interactivity add "use client"
  • No useEffect data fetching — data is imported at module level
  • Admin pages load mock data into local useState, then mutate state locally (no persistence)
  • No global state library — all state is local to the page component

🔐 Middleware & Auth

src/middleware.ts

Runs on every request matching /admin/:path*. Allows /admin/login without a session. Checks for the cookie admin_session=authenticated and redirects to /admin/login if missing.

API Route — /api/admin/auth

// POST /api/admin/auth
// Login
{ action: "login", username: "admin", password: "diamond123" }
// → sets cookie: admin_session=authenticated (httpOnly, path=/)

// Logout
{ action: "logout" }
// → clears cookie
⚠️
Credentials are hardcoded (admin / diamond123). This is a prototype-only authentication mechanism — not production-safe.

🔬 Mock Data — Generation Strategy

All mock data is generated at module load time using a seeded LCG (Linear Congruential Generator) RNG. The same seed always produces the same output, making the data deterministic and reproducible across sessions.

class RNG {
  private s: number;
  constructor(seed: number) { this.s = ((seed >>> 0) || 1) & 0x7fffffff; }
  next(): number {
    this.s = (this.s * 1664525 + 1013904223) & 0x7fffffff;
    return this.s / 0x7fffffff;
  }
  int(n: number) { return Math.floor(this.next() * n); }
  bool(p = 0.5) { return this.next() < p; }
  pick<T>(a: T[]) { return a[this.int(a.length)]; }
}

Date-Relative Generation

All ranges are computed from new Date() at module load. Bookings cover 30 days before today through 90 days ahead. Schedule availability covers 13 weeks forward. This ensures the prototype never shows stale dates regardless of when it is opened.

Generation Details

  • Range: 30 days before today → 90 days ahead (~3 months)
  • Volume: ~30 bookings/day on weekdays (7% slow days: 8–17, 7% packed: 42–49, Saturday: 32–45, Sunday: 18–31)
  • 90 recurring series placed first; single-session bookings fill the remaining target
  • Lane grid: 4 lanes × 24 slots. tryAssignLane(si, numSlots, instId) checks both lane AND instructor occupancy — an instructor cannot appear in two lanes at the same time
  • Status by date: past → 76% completed / 18% cancelled / 6% no_show; today → time-aware; future → 88% confirmed / 12% cancelled
  • Instructor pool: i1, i2, i3, i5, i6, i7, i8, i9, i10, i11 (10 active instructors)
  • Customer pool: 150 customers (c1–c150)

📨 Data — Notifications

type NotificationRecord = {
  id: string;
  bookingId: string;
  bookingReference: string;
  recipientType: "customer" | "instructor" | "admin";
  recipientName: string;
  recipientEmail: string;
  notificationType: "confirmation" | "reminder_24hr" | "reminder_2hr_sms"
    | "change" | "cancellation" | "calendar_invite";
  channel: "email" | "sms" | "calendar";
  sentAt: string;
  deliveryStatus: "sent" | "pending" | "failed";
};

Generation Rules

  • Window: last 30 days only (no future notifications)
  • Cap: maximum 300 records — all 6 notification types guaranteed to appear
  • Per booking: confirmation (email), calendar invite (instructor), 24hr reminder (email), 40% chance of 2hr SMS
  • Cancelled bookings also get: cancellation email to customer + instructor
  • 5% of bookings get an admin "change" notification
  • Sorted by sentAt descending (most recent first)
