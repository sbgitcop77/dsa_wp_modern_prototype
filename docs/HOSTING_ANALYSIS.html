1. Clarifying what "Cloudflare" actually means here

The original concern was that "Cloudflare is just a front for websites hosted on other platforms." That's true of one Cloudflare product, but not the one this project uses.

ProductWhat it actually is
Cloudflare CDN / proxy
(the "orange cloud")		 Sits in front of a site hosted elsewhere (a VPS, WordPress host, AWS, etc.) for DDoS protection, caching, and free SSL. The real site lives on a separate origin server. This is the product the "just a front" description fits.
Cloudflare Pages
(what this project uses)		 A real hosting platform — the same category as Vercel, Netlify, or GitHub Pages. There is no separate origin: Cloudflare's network is the host. Configured here via wrangler.toml and next.config.mjs's output: export.
The "just a front for other hosts" concern doesn't apply to what's actually deployed in this repo. The relevant comparison is Cloudflare Pages vs. other static hosting platforms — not Cloudflare vs. "real" hosting.

2. Could this be hosted on WordPress instead?

This project is Next.js / React / TypeScript. WordPress is PHP + MySQL. WordPress cannot run this codebase directly — "hosting on WordPress" means rebuilding, not just redeploying. Three real options:

Option 1 — Rebuild natively in WordPress

Recreate the design as a custom PHP theme; replace the bespoke admin/booking logic with WordPress plugins (e.g. Amelia or Bookly for booking, ACF for content, WooCommerce for payments/memberships).

Pros
  • Non-technical staff can edit content via wp-admin
  • Huge plugin ecosystem
  • Matches the real business's existing site (thediamondsportsacademy.com)
Cons
  • Discards nearly everything built so far — booking wizard, recurring-series conflict validation, lane-utilization heatmap, waitlist auto-promotion, CSV import wizard
  • None of those are standard WP booking-plugin features — would require custom PHP plugin development from scratch, a different stack entirely

Option 2 — Headless WordPress + keep the Next.js frontend

WordPress becomes a content backend only (pages, coach bios, blog) exposed via REST/WPGraphQL; this React frontend keeps rendering everything.

Pros
  • Non-technical content editing without losing the current UI/booking-wizard work
Cons
  • Two systems to maintain (WP backend + Next.js frontend) instead of one
  • Next.js still needs a Node-capable host regardless — WordPress hosting alone can't serve it
  • Bespoke booking/admin logic still needs a real backend either way

Option 3 — Stay static (current path), add a real backend later

Keep the React/Next.js prototype as-is; when ready for production, add Cloudflare Workers + D1 (or an external API) for real bookings/auth/payments instead of mock data.

Pros
  • Nothing already built gets discarded
  • Cheapest hosting, global CDN, no PHP/MySQL security surface
  • Fits a static prototype with no real DB perfectly as-is
Cons
  • No built-in editorial CMS — content changes require a code change + redeploy

3. Does switching to Vercel reduce vendor lock-in?

No — this was a misconception worth correcting. The project already made the most portability-friendly choice available: output: export in next.config.mjs produces plain static HTML/CSS/JS in /out. That build output is framework- and host-agnostic — it can be deployed to Cloudflare Pages, Vercel, Netlify, GitHub Pages, S3+CloudFront, or a bare Nginx box with zero code changes.

The only Cloudflare-specific artifact today is wrangler.toml — one small config file. Removing it and deploying the same /out build to Vercel or Netlify is a 10-minute change, not a rebuild.

The real lock-in risk isn't which static host is chosen today — it's adopting a platform's proprietary runtime services later for the real backend this prototype will eventually need (everything is still mock data today). Vercel's ISR/Edge Config/KV, Cloudflare's Workers + D1 + Durable Objects, and Netlify's Functions/Blobs are all equally capable of creating lock-in once real backend logic depends on them. That risk is symmetric across all three providers — Vercel is not inherently "more portable."

Key finding: Vercel's free Hobby tier explicitly prohibits commercial use. Since this is a business website, Vercel would likely require the $20/month-per-seat Pro plan from day one — while Cloudflare Pages and Netlify both permit commercial use on their free tiers.

5. Bottom line (prototype-stage findings)

  • The "Cloudflare is just a front for other hosts" concern applies to Cloudflare's CDN/proxy product, not Cloudflare Pages, which is what this project actually uses.
  • The current static-export setup is already close to maximally portable, independent of which host is chosen — switching between Cloudflare Pages, Vercel, and Netlify today is a small config change, not a rebuild.
  • Cloudflare Pages is currently the cheapest of the three options for this commercial, low-to-moderate-traffic site, mainly due to unlimited free bandwidth and no commercial-use restriction.
  • No strong technical or cost case was found for moving off Cloudflare Pages at this time.

6. Production cost comparison — live site with a real backend

Security gap to budget for regardless of platform choice: the live site shouldn't ship with the current hardcoded admin / diamond123 cookie protecting customer data. The fix is effectively free either way — Cloudflare Access (free for up to 50 users, protects /admin/* with zero app code) fits naturally if Cloudflare is chosen, or NextAuth (free, just dev time) works on any platform. This doesn't move the monthly cost figures above, but it's necessary dev work before going live.

7. Final recommendation — production stack

  • Cloudflare: full stack (hosting + DB + functions) effectively $0/mo at this traffic level, with the entire D1 free tier as buffer. Add ~$3–4/mo for SMS. Comfortably under the $25/mo budget with room to grow before hitting any paid tier.
  • Vercel: hosting alone is $20/mo per seat — over 80% of the budget gone before the database, email, or SMS are even counted. Cost multiplies if more than one person needs platform access.
  • Netlify: technically $0/mo, but hosting bandwidth and database usage draw from the same 300-credit pool, making it the most likely of the three to unexpectedly tip into the $20/mo Pro tier — especially once Netlify DB storage stops being free in July 2026.
  • Recommendation: Cloudflare is the clear winner for this traffic/budget combination — not just slightly cheaper, but the only one of the three with real margin left in a sub-$25/mo budget after the database and functions are accounted for.
